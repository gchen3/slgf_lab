# Website Implementation Plan

This plan turns the current Quarto website into a reliable, publishable SLGF-Lab site. The immediate goal is to stabilize rendering and navigation before investing in presentation improvements.

## Current Status

- The project is a Quarto website with GitHub Pages publishing configured in `.github/workflows/quarto-publish.yml`.
- `index.qmd` is the primary homepage and contains the active SLGF-Lab overview and current research project summaries.
- `about.qmd` is still placeholder content and includes a sample R code chunk.
- The former `index_quarto.qmd` draft page has been removed so the site has one clear homepage source.
- `styles.css` is effectively empty.
- Generated Quarto output and processing artifacts are ignored so the repository focuses on source files.
- Local rendering currently fails because Quarto cannot execute required local tools on this machine.

## Recommended Sequence

1. Stabilize the source pages so the website can render without unnecessary R execution.
2. Replace placeholder content on the About page.
3. Convert homepage placeholder project links into working targets or remove them until target pages exist.
4. Add a small, maintainable visual layer through `styles.css`.
5. Regenerate `_site` and confirm search metadata reflects the current content.
6. Verify the GitHub Pages workflow and publish from `main`.

## Implementation Notes

- Keep the first pass simple: static Quarto pages should render before adding interactive R widgets.
- Avoid adding package-dependent R chunks unless the workflow explicitly installs those packages in GitHub Actions.
- Treat `_site` as generated output. Source files should be the primary place for edits, and `_site` should be regenerated by Quarto during publishing.
- Fix local Quarto/R execution separately from content changes if the issue is machine permissions rather than project configuration.
